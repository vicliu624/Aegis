# Aegis

[English](./README.md)

> 面向 MCU 系统的原生应用运行时与受治理设备平台。

## 什么是 Aegis？

**Aegis** 是一个面向 MCU 系统的原生应用运行时，以及一个受治理的设备平台。

它建立在一个简单原则之上：

- **系统**拥有硬件，
- **系统**治理设置、服务与生命周期，
- **应用**则作为独立构建的原生扩展，在运行时被发现、加载、运行并卸载。

Aegis 不是传统意义上的桌面操作系统，
也不只是一个简单的启动器。

它是一个面向资源受限设备的受控运行时。

---

## 当前主线

这个仓库当前有一条应当优先遵循的主线：

- 基于 Zephyr 的常驻系统 bring-up
- 面向 LilyGo T-LoRa Pager 的硬件适配路径
- 通过 appfs 打包并加载原生运行时应用
- 通过 [scripts/build_zephyr.py](./scripts/build_zephyr.py) 作为顶层构建与刷写入口

如果你是第一次进入这个仓库，请把上面这条 Zephyr 路线视为当前权威构建路径。

## 如何开始

如果你的目标是在一台新机器上把当前 Zephyr 目标跑起来，请按下面顺序走：

1. 先运行主机环境自检。
2. 再阅读快速上手文档。
3. 如果自检提示缺依赖，再完成环境 onboarding。
4. 然后通过顶层构建驱动完成 configure、build 和 flash。

### 第一条命令

Windows：

```powershell
python scripts/build_zephyr.py check-host-env
```

Linux、macOS 或 WSL：

```bash
python3 scripts/build_zephyr.py check-host-env
```

如果你需要给自动化脚本或外层工具使用的机器可读输出：

Windows：

```powershell
python scripts/build_zephyr.py check-host-env --json
```

Linux、macOS 或 WSL：

```bash
python3 scripts/build_zephyr.py check-host-env --json
```

## 为什么需要 Aegis

大多数 MCU 固件都是以单一的整体镜像形式构建的。

随着时间推移，每一个界面、工具、实验功能、协议分支和实用模块，都会被编译进同一个二进制里。结果通常也很熟悉：

- 固件不断膨胀，却缺少清晰边界，
- 系统代码和应用代码逐渐纠缠，
- 硬件访问四处泄漏，
- 生命周期越来越难以推理，
- 每增加一个功能，都像是在重新触碰整个世界。

Aegis 选择了另一条路线。

它把设备看作一个**被治理的小世界**：

- 有一个**常驻系统**持续保持控制权，
- 硬件由系统集中管理，
- 服务通过明确接口对外暴露，
- 应用则以存放在文件系统中的**可加载原生单元**形式存在。

启动时，Aegis 会扫描应用目录，构建应用目录清单，并将可用应用展示给用户。
当用户选择某个应用时，Aegis 会加载它、把它绑定到主机 API、创建其会话，并为它提供一个受控的运行空间。
当应用退出后，Aegis 会将其拆除并回收相关资源。

这不是“带很多页面的固件”。
这是一个**支持原生应用的设备运行时**。

## 核心理念

### 硬件属于系统

驱动、板级资源和设备协同始终由系统掌控。
应用消费的是能力，而不是对底层资源的原始所有权。

### 应用是原生的，但不是主权实体

应用可以是独立构建的原生二进制，但它们仍然是运行时里的访客。
它们不定义这台机器，
而是在这台机器之内运行。

### 运行时加载是一等概念

应用从存储中被发现，而不是被硬编码进主固件菜单。
安装状态与存在性由文件和元数据来表达。

### 生命周期是显式的

加载、启动、活跃会话、退出、拆除、卸载与恢复，都不是偶然行为。
它们本身就是架构的一部分。

### 系统与应用必须保持区分

系统是长期存在的权威主体，
应用则是可替换的工作负载。

## 高层结构

Aegis 主要围绕以下部分组织：

- **Zephyr Base**：操作系统基底、驱动、文件系统、设置、调度、内存、电源
- **Aegis Core**：常驻系统权威核心
- **Aegis Services**：系统能力的受控服务网关
- **Aegis Runtime**：原生应用加载与会话运行时
- **Aegis Shell**：启动器、主页、设置与状态界面

---

## Aegis 与现有方案的对比

Aegis 并不是凭空出现的。
它靠近若干已有的嵌入式思路，但又不等同于其中任何一种。

### Zephyr + LLEXT

这是与 Aegis 最接近的技术基础。

Zephyr 的 LLEXT 已经提供了在运行时加载独立构建原生扩展的底层能力。
这意味着它解决了问题中非常关键的一部分：**原生代码的运行时加载**。

但仅有 LLEXT 仍然只是一个机制。
它本身并不定义：

- 应用身份，
- manifest 结构，
- 目录发现，
- 能力契约，
- 主机 API 语义，
- 生命周期治理，
- 设备适配规则，
- 以及系统 shell 行为。

Aegis 做的，是把这个底层机制提升为一个**完整的设备应用平台**。

### NuttX 可加载模块

NuttX 展示了另一条相近路径：嵌入式系统也可以支持外部可加载的程序单元。

这一点与 Aegis 有一个关键相似处：
应用不必永远被熔进主固件镜像里。

但两者的架构重心不同。

NuttX 更像是：

> 一个可以运行独立链接程序的小型操作系统

而 Aegis 更像是：

> 一个由系统掌权、以结构化访客形式接纳应用的受治理设备运行时

差别不仅在技术层面，
更在于是谁拥有这个设备世界的控制权。

### Tock

如果从隔离、权力边界和嵌入式系统中的受控执行来看，Tock 是一个非常相关的参考。

它有助于理解：系统如何保持主导，而应用如何保持受约束。

但 Tock 的重心更偏向：

- 内存安全，
- 保护边界，
- 内核/用户态结构，
- 以及安全隔离。

Aegis 的重心则不同：

- 原生运行时加载，
- 面向产品的应用生命周期，
- 由文件系统承载的应用存在性，
- 受能力治理的服务接口，
- 以及一个常驻系统 shell。

所以从理念上看，Tock 与 Aegis 相邻，但并不是同一种产品形态。

### 嵌入式 WebAssembly 运行时

面向嵌入式设备的 Wasm 运行时，在精神上也与 Aegis 很接近。

它们探索的是：

- 可移植的应用模块，
- 主机提供的 API，
- 执行边界，
- 以及运行时可扩展性。

这与 Aegis 在概念上有重叠。

但 Aegis 选择的是另一条主路径：

- **优先原生扩展**，
- 而不是优先虚拟指令格式。

从这个角度看，Wasm 平台是一条通往可扩展性的相邻路线，而 Aegis 走的是原生运行时路线。

## Aegis 的独特之处

Aegis 的独特性，并不在于“动态加载”这个概念本身是新的。
动态加载早已以多种形式存在。

真正独特的是这些理念的**组合方式**，以及其背后的架构立场。

### 1. Aegis 不只是一个加载器

加载器只能把代码带进内存，
而 Aegis 做得更多。

它围绕已加载应用定义了一个受治理的运行时：

- 发现，
- 应用编目，
- 元数据解释，
- 主机绑定，
- 会话创建，
- 受控执行，
- 退出处理，
- 拆除，
- 卸载，
- 以及恢复。

在 Aegis 中，加载只是更大应用生命周期中的一个阶段。

### 2. Aegis 不是“带插件的固件”

很多系统会使用 plugin 这个词，但这些插件最终仍然会直接依赖板级细节、驱动假设和私有系统内部实现。

Aegis 拒绝这种模式。

应用不应该沦为板级代码的隐性延伸，
它们应该保持为应用本身。

这意味着系统与应用之间的边界不是偶然形成的，
而是架构层面的明确设计。

### 3. Aegis 将硬件所有权保留在系统中

这是它最重要的差异之一。

在 Aegis 中：

- 系统拥有硬件，
- 系统负责服务中介，
- 系统治理设置，
- 应用通过能力来消费服务，而不是直接攫取底层资源。

这会改变“可扩展性”的含义。

应用足够强大，可以做有价值的事情，
但又不会拥有重新定义整台机器的主权。

### 4. Aegis 为硬件多样性而设计

Aegis 假设了一种真实存在的嵌入式环境：

- 不同的屏幕，
- 不同的输入方式，
- 不同的无线电，
- 不同的传感器，
- 不同的存储布局，
- 不同的内存预算，
- 不同的板卡。

许多扩展系统恰恰会在这里变得脆弱。
板级差异不断向上泄漏，最终每个应用都变成一次硬件协商。

Aegis 希望通过以下方式吸收这种差异：

- 设备适配，
- 能力映射，
- 以及由主机治理的接口。

这对于构建一个真正可跨多设备运行的 MCU 应用平台至关重要。

### 5. Aegis 将运行时可扩展性提升为产品架构

许多现有机制只是技术特性，
而 Aegis 试图把可扩展性提升为一种**系统级架构原则**。

这意味着，设备会被理解为：

- 一个常驻世界，
- 拥有长期权威主体，
- 具备受控服务，
- 具有显式生命周期，
- 并承载可替换的原生工作负载。

这才是更深层的转变。

Aegis 问的不是：
“MCU 代码能不能动态加载？”

它问的是：
“当运行时加载的应用被视为一等架构公民时，嵌入式设备会变成什么？”

---

## 一句话概括

如果说 Zephyr LLEXT 提供的是底层加载机制，
那么 Aegis 想要提供的，是建立在其之上的**完整受治理应用平台**。

---

## Aegis 不是什么

Aegis **不是**：

- 面向微控制器的桌面操作系统，
- 类 Linux 的包管理环境，
- 一个允许无限制硬件访问的松散插件模型，
- 一个伪装成应用平台的菜单系统，
- 也不是一个随着时间不断堆页面的整体式固件。

Aegis 是一个**带原生应用的受治理 MCU 运行时**。

## 仓库结构

```text
/core        常驻系统核心
/runtime     原生应用运行时与加载器集成
/services    系统能力的服务网关
/shell       启动器、设置、主页与可见系统界面
/sdk         Aegis 应用的头文件与构建支持
/apps        示例应用或内建开发应用
/docs        架构说明与设计文档
```
